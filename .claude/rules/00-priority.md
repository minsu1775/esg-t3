# 규칙 우선순위 시스템

규칙들이 충돌할 때 아래 우선순위를 따른다. 낮은 우선순위 규칙이 높은 우선순위 규칙을 위반하는 방향으로 적용되어선 안 된다.

## 우선순위 레벨

| 레벨 | 이름 | 대표 규칙 | 타협 가능 여부 |
|---|---|---|---|
| **P0** | 테넌트 격리 + 감사 무결성 | RLS 필수, AuditLog INSERT-only, Hash Chain 정합 | **절대 불가** |
| **P1** | 재현성 | 배출량 계산 원천 데이터 INSERT-only, FormulaVersion 기록 | 불가 |
| **P2** | 성능 | N+1 방지, 인덱스 전략, 배치 처리 | 조건부 완화 가능 |
| **P3** | 개발자 편의 | 명명 규칙, 코드 포맷, Lombok 사용 | 유연하게 적용 |

## 충돌 시 판단 기준

**P0 예시 — 절대 우선**:
- "RLS 우회하면 N+1 없어진다" → P0 > P2. RLS 우회 금지. 인덱스 최적화로 P2 해결.
- "AuditLog UPDATE가 편하다" → P0 > P3. INSERT-only 유지.
- "테넌트 필터 빠뜨리면 코드가 단순하다" → P0 > P3. 필터 필수.

**P1 예시 — 재현성 보호**:
- "최신 값만 필요하니 버전 관리 생략" → P1 위반. 항상 버전 컬럼 유지.
- "빠른 수정을 위해 UPDATE 사용" → P1 위반. 정정은 새 버전 INSERT.

**P2 예시 — 조건부 완화**:
- 복잡한 Join이지만 N+1보다 나은 경우: `@EntityGraph` 또는 `JOIN FETCH`로 해결.
- 배치 처리 성능이 중요한 경우: Virtual Threads + `REQUIRES_NEW` 병행 적용.

**P3 예시 — 유연 적용**:
- 명명 규칙 예외: 레거시 API 연동 시 서드파티 필드명 유지 허용.
- Lombok 금지 목록 외 사용: 팀 합의 후 적용 가능.

## 코드 리뷰 체크포인트

- P0 위반 → **즉시 차단** (머지 불가)
- P1 위반 → **차단** (재현성 리스크 문서화 시 예외 가능)
- P2 위반 → **경고** (성능 테스트 결과 첨부 시 머지 가능)
- P3 위반 → **권고** (팀 합의 시 예외 가능)
